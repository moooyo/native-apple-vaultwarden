# Tessera

A from-scratch, **pure-Swift, byte-compatible Bitwarden/Vaultwarden client** for iOS and macOS, built with the macOS 26 / iOS 26 **Liquid Glass** design language. Tessera re-implements Bitwarden's client-side cryptography and sync protocol natively (no Rust SDK), talks to a self-hosted [Vaultwarden](https://github.com/dani-garcia/vaultwarden) server, and ships a system-wide AutoFill + passkey extension.

> Tessera is *compatible with Bitwarden®* but is an independent project; it is not affiliated with or endorsed by Bitwarden, Inc.

## Status

M1 is in **signed-device / real-server validation**. The Swift package contains 15 library
products and 14 executable test runners. On 2026-07-16 the final review tree passed all 14
runners (**1305 checks**), `swift build`, `xcodegen generate`, and unsigned Xcode 27 beta 2
builds of both app schemes (including both AutoFill targets).

The current implementation includes:

- An encrypted cache backed by the official
  [`SQLCipher.swift`](https://github.com/sqlcipher/SQLCipher.swift) **4.17.0** package (exactly
  pinned in `Package.swift` and `Package.resolved`), including validation that SQLCipher is really
  loaded and a guarded migration from the earlier plaintext system-SQLite cache.
- Read, sync, offline-outbox, edit, and display support for all five Bitwarden cipher types:
  **Login, Secure Note, Card, Identity, and SSH Key**. Item keys and passkey/custom-field data are
  preserved on edits.
- Password, TOTP/one-time-code, and passkey AutoFill paths, including exact binary passkey-ID
  matching, concrete system identities, and the OTP completion API. Direct fulfillment decrypts
  only the selected cipher; manual-picker discovery scans a bounded, account-scoped set of login
  rows and decrypts only display metadata until the user selects a credential. The real picker
  routes password, OTP, and allowed-credential-filtered passkey selections to their distinct
  AuthenticationServices completion APIs.
- Canonical account IDs bind access-token leases, refresh/local-auth Keychain records, biometric
  keys, CRUD, and outbox processing. Account-composite entity keys allow cloned servers to reuse
  cipher/folder UUIDs safely; the migration preserves unowned encrypted payloads in quarantine
  and never guesses ownership for legacy `host|email` rows. API/repository generations plus a
  random cross-process session nonce reject stale A→B→A work. Cold restoration preserves the
  server/KDF metadata needed to unlock, while AutoFill identities are authoritatively replaced
  and stale publications are revoked across account transitions.
- A durable `PasskeyHandoff` stages a UUID-only App Group marker, stores the PKCS#8 secret in the
  shared Keychain, and atomically promotes the marker when ready. It recovers stale staging work,
  cancels system registration if staging fails, verifies the expected account during main-app
  import, and acknowledges only after the local row plus its receipt-linked outbox state commits
  atomically. Sync finalization atomically replaces placeholders, completes linked receipts, and
  clears outbox rows so a stale drainer cannot recreate a server item.
- iOS, macOS, and AutoFill targets generated by XcodeGen; the current review tree builds for
  macOS and a generic iOS device with signing disabled.

## Architecture

The code is grouped into five responsibility bands with deliberately narrow SwiftPM target
dependencies. Security-sensitive code lives in L0/L1. The **AutoFill extension directly depends
only on a minimal subset** (`VaultReader + KeychainBridge + VaultModels + Fido2 + DesignSystem +
AppShared + PasskeyHandoff`); their transitive read/crypto stack includes `CryptoCore`, `KeyVault`,
`VaultStore`, and lightweight `Generators`. It never links networking, sync, the repository, or
the main UI, which keeps it suitable for the system's roughly 120 MB extension memory budget.

```
L4  App         App-iOS · App-macOS · AutoFillExtension
L3  UI          UI-iOS · UI-mac  ←shared←  UIShared(@Observable) · DesignSystem
L2  Services    Networking · SyncEngine · Generators · VaultRepository · AppShared
L1  Security    KeyVault · KeychainBridge · VaultStore · VaultReader · Fido2 · PasskeyHandoff
L0  Spine       CryptoCore · VaultModels
```

The unwrapped User Key lives only inside the `KeyVault` actor in memory and is zeroized on lock; it crosses to the extension only as a Secure-Enclave-ECIES-wrapped blob behind biometrics.

## Key decisions

- **Pure-Swift crypto** (CryptoKit + CommonCrypto + Security) — the Bitwarden Rust SDK is license-incompatible with an MIT/App-Store client.
- **PBKDF2-only** — Argon2id accounts are rejected at login with a clear message (intentional scope limit; removes the AutoFill-extension Argon2 OOM risk).
- **Vaultwarden self-hosted** is the official target; a custom server URL is supported.
- **No push** (self-hosted gets no APNs) — polling + `BGAppRefreshTask`/`NSBackgroundActivityScheduler` + revision-token incremental sync; unknown EncString types soft-fail instead of aborting a sync.
- Crypto is pinned to **golden vectors** (PBKDF2/HKDF/AES-CBC+HMAC/TOTP/WebAuthn) generated and verified during development.

## Build & test

**Libraries and headless checks:**
```sh
swift build                       # compiles all 15 library products
swift run CryptoCoreTests         # each package has an executable test runner
swift run VaultRepositoryTests    # ... (XCTest is unavailable on CLT; see docs)
```
(The repository uses `.executableTarget` runners rather than `swift test`; run all 14
`<Module>Tests` executables, with exit 0 meaning success.) The first build resolves the pinned
SQLCipher package.

**App + extension — requires full Xcode with the iOS/macOS 26 SDK:**
```sh
brew install xcodegen
xcodegen generate                 # produces Tessera.xcodeproj from project.yml
open Tessera.xcodeproj
# set DEVELOPMENT_TEAM, replace the placeholder bundle ids / App Group, then build
#   the Tessera-iOS or Tessera-macOS scheme.
```
See [`App/README.md`](App/README.md) for assembly details.
